## 一、背景

当前策略卡和亲密度只能通过管理员手动设置。OC 不会自己学习「该怎么对待这个人」，也不会自动感知和谁更亲近。

## 二、目标

| 编号 | 内容 | 说明 |
|------|------|------|
| P3-反思自动策略 | 每次反思（每6轮对话）时，额外分析本轮对话中对用户的行为调整建议，自动更新策略卡 | 扩展现有反思管线 |
| P3-亲密度自动计算 | 每次反思时，基于互动频率、对话深度、情感记忆密度、时间衰减，自动更新亲密度 | 扩展现有反思管线 |

## 四、亲密度自动计算

### 4.1 计算公式

```
亲密度 = clamp(
    基础分 * 0.3 + 互动频率 * 0.25 + 情感密度 * 0.25 + 对话深度 * 0.2
  - 时间衰减
, 0.0, 1.0)
```

### 4.2 各因子取值

| 因子 | 来源 | 计算方式 |
|------|------|----------|
| 基础分 | 当前亲密度 | 如果已手动设置过，取当前值作为基础分 |
| 互动频率 | `_reflection_states` 中的交互轮数统计 | 本次为 1 轮，累加到用户维度（需新增计数器） |
| 情感密度 | 新增 SQL 查询 | `COUNT(*) WHERE memory_type='emotional' AND tag LIKE userId` |
| 对话深度 | chat_records 文本长度 | 本轮用户消息总字数 / 500 归一化 |
| 时间衰减 | 上次交互时间 | `距离上次 * 0.01`，7 天即衰减 0.1 |

### 4.3 最小值保护

- 亲密度一旦达到某个值，不会回退到太远（最小保留 50%）
- 手动设置的值在自动计算时作为基础分优先保留

### 4.4 新增 SQL

在 `memory_sql_manager.py` 中新增 `count_memories_for_user(user_id_tokens, memory_type=None, scope="public")`：

```sql
SELECT COUNT(DISTINCT mr.id)
FROM memory_records mr
JOIN memory_tag_rel mtr ON mtr.memory_id = mr.id
JOIN global_tags gt ON gt.id = mtr.tag_id
WHERE gt.name IN (user_id_tokens)
  AND mr.strength > 0
  AND mr.memory_scope = ?
  [AND mr.memory_type = ?]   -- 可选：筛选情感记忆
```

（待确认：如果 SQL 查询太复杂或性能堪忧，则改用「从缓存记忆列表中统计」的轻量方案）

### 4.5 更新方式

- **渐进更新**：每次反思后小幅调整（±0.05），状态平滑过渡
- 调完后调用 `strategy_store.set_intimacy(user_id, score)`

## 五、反思自动策略

### 5.1 流程

```
反思执行 → 记忆生成完毕
  ↓
  （新增）构建策略分析 prompt
  ↓
  调用 analyzer_model（小模型，同反思模型）
  ↓
  解析输出 → 提取策略建议文本
  ↓
  调用 strategy_store.set_strategy(user_id, strategy, source="反思自动总结 | 日期")
```

### 5.2 与反思的关系

- **复用同一个模型调用**（analyzer_model），不在反思之外再发一次请求
- 具体做法：在反思的 prompt 末尾追加一段「同时分析相处策略」，让 LLM 一次返回两个结果（feedback + strategy）

### 5.3 Prompt 追加内容

```
## 相处策略分析
基于以上聊天内容，判断你是否需要调整与以下用户的相处方式：
- 输出格式：{ "strategies": [ {"user_id": "...", "strategy": "..."} ] }
- 如果不需要调整，输出空数组
- 策略应简洁（一句话），不要冗长
```

### 5.4 输出样例

```json
{
  "strategies": [
    {"user_id": "1432192649", "strategy": "和 T-White 相处时保持轻松自然不刻意"}
  ]
}
```

### 5.5 策略更新逻辑

- 如果 LLM 返回了策略 → 调用 `strategy_store.set_strategy()`
- 策略来源标记为 `反思自动总结`
- 如果该用户已有管理员手动设置的策略（source 包含「管理员指令」），则不覆盖

## 八、测试重点

1. **反思日志** — 确认反思结束后出现额外的策略/亲密度更新日志
2. **WebUI 策略卡** — 反思后用户画像页应有自动生成的策略卡，来源标记为「反思自动总结」
3. **WebUI 亲密度** — 反思后亲密度数值自动变化
4. **管理员手动策略不被覆盖** — 手动设置的策略在自动反思时不会被替换
5. **基础反思不受影响** — 即使自动策略部分报错，记忆生成仍正常

## 九、风险

| 风险 | 应对 |
|------|------|
| 新增 SQL 查询性能 | 使用索引 `memory_tag_rel.tag_id + memory_id`，单次查询通常 < 10ms |
| LLM 输出不稳定 | JSON 解析失败 → 自动跳过，不影响反思 |
| 反思间隔内多用户 | 对每个出现在 chat_records 中的用户都计算一次 |
| 亲密度波动过大 | 渐进更新（每次 ±0.05），平滑过渡 |
